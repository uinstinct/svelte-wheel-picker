  <action>
Modify `.github/workflows/release.yml` so the release job:

1. Still triggers on push to `main` and `workflow_dispatch`.
2. After computing the new version, creates a branch named `releases/v{NEW_VERSION}` from the current commit.
3. Commits the `package.json` version bump to that branch and pushes it (NOT to main).
4. Creates the GitHub Release tag pointing at that release branch commit.
5. The `publish` job checks out the release branch (using the tag `v{NEW_VERSION}`) instead of `ref: main` so it gets the updated `package.json` with the correct version.

Key changes to the `release` job:
- After "Bump version" step, replace the "Commit and push version bump" step with:
  ```
  BRANCH="releases/v${{ steps.newver.outputs.version }}"
  git checkout -b "$BRANCH"
  git add package.json
  git commit -m "chore: bump version to ${{ steps.newver.outputs.version }}"
  git push origin "$BRANCH"
  ```
- In the "Create GitHub Release" step, add `--target "$BRANCH"` so the tag targets the release branch.

Key changes to the `publish` job:
- Change `ref: main` to `ref: v${{ needs.release.outputs.version }}` (checkout by tag).
- Add an `outputs` section to the `release` job: `outputs: version: ${{ steps.newver.outputs.version }}`

Remove the `[skip ci]` from the commit message since the commit is no longer on main (no infinite loop risk).
