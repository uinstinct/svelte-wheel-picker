## Component Styling Contract

### WheelPickerWrapper DOM Structure

The wrapper renders `<div data-swp-group>` with an optional consumer class via `classNames.group`. For the Phase 5 demo, the wrapper receives `classNames={{ group: 'time-picker-group' }}`.

**Demo CSS for the new WheelPickerWrapper example:**

```css
/* Applied in +page.svelte <style> block — consumer CSS using data attributes */

:global([data-swp-group].time-picker-group) {
  display: flex;
  flex-direction: row;
  align-items: stretch;
}
```

No gap between wheels — they sit directly adjacent. The `.wheel-container` provides the surrounding padding context.

### Focus Ring

Each `WheelPicker` wrapper div has `tabindex="0"`. When focused via keyboard Tab, the browser's default focus ring applies. No custom focus ring is specified — the default outline is sufficient for the demo page and aligns with the headless philosophy (consumers style focus as needed).

### Disabled Option Opacity

`opacity: 0.3` on `[data-swp-option][data-swp-disabled='true']`. Unchanged from Phase 3. Source: `+page.svelte` line 162–164.

## Interaction Contract

### Tab/Focus Navigation Between Wheels

- Tab moves focus from the first `WheelPicker` wrapper to the second in DOM order.
- Shift+Tab moves focus in reverse.
- Focus is managed entirely by native DOM tab order — no JavaScript focus management.
- Visual feedback: browser default focus outline on the focused wheel's `tabindex="0"` div.
- Source: CONTEXT.md D-04; RESEARCH.md "Focus management between wheels" anti-pattern section.

### Within-Wheel Keyboard Navigation

Unchanged from Phase 3: Arrow Up/Down moves one item; Home/End jumps to first/last enabled option. Each wheel responds independently. Source: REQUIREMENTS.md INTR-05.

### Touch/Mouse Interaction

Unchanged from Phase 3: pointer drag with inertia, scroll wheel with sensitivity. Each wheel in a wrapper is fully independent. Source: REQUIREMENTS.md INTR-01, INTR-02, INTR-03.
