**Primary recommendation:** Build WheelPickerWrapper first (single commit), then do the barrel export trim + types extension in a second task, then verify with `npm pack` dry-run and a manual SSR import test. Both tasks are self-contained and testable in isolation.

## Architecture Patterns

### WheelPickerWrapper Structure

The wrapper is a thin layout container only. It has no reactive state, no lifecycle, no event handling. Its entire job is to provide a named DOM group and apply a consumer-supplied class.

```svelte
<!-- src/lib/WheelPickerWrapper.svelte -->
<script lang="ts">
  import type { WheelPickerWrapperProps } from './types.js';
  import type { Snippet } from 'svelte';

  let { classNames, children }: WheelPickerWrapperProps & { children?: Snippet } = $props();
</script>

<div data-swp-group class={classNames?.group ?? undefined}>
  {@render children?.()}
</div>
```

**Key pattern notes:**
- `Snippet` is imported from `'svelte'` (not a sub-path). Svelte 5 provides this type at the root.
- `children` in Svelte 5 is a special reserved prop name that receives the component's default slot content as a `Snippet`. No import of `children` from Svelte â€” it arrives as a prop.
- `{@render children?.()}` handles the case where no children are passed without runtime error.
- No `<style>` block â€” headless by design.
- No `onMount`, no `$effect` â€” no browser API usage at any scope.

### Type Pattern (following WheelPickerClassNames)

```typescript
// In src/lib/types.ts â€” extend the existing type file

export type WheelPickerWrapperClassNames = {
  /** Outer group container div */
  group?: string;
};

export interface WheelPickerWrapperProps {
  /** Per-element CSS class overrides for the wrapper group. */
  classNames?: WheelPickerWrapperClassNames;
}
```

The `children` prop is NOT added to `WheelPickerWrapperProps` because Svelte 5 injects `children` as a built-in snippet prop â€” declaring it in the Props interface causes type conflicts. The component file uses an intersection type: `WheelPickerWrapperProps & { children?: Snippet }`.

### Anti-Patterns to Avoid

- **Declaring `children` in WheelPickerWrapperProps:** Svelte 5 treats `children` as a built-in snippet prop. Adding it to the Props interface causes TypeScript errors on the component definition.
- **Using `<slot>` instead of `{@render children?.()}`:** Svelte 5 uses the snippet API. `<slot>` still works in Svelte 5 but is considered legacy and will not interop cleanly with the runes-based component model. Use `{@render children?.()}`.
- **Calling browser APIs in WheelPickerWrapper constructor/module scope:** No physics, no RAF, no event bindings needed â€” there is nothing to protect against, but do not add any.
- **Re-exporting hooks as "convenience exports":** D-07 is explicit: hooks are implementation details. Consumers have no valid use case for `useControllableState` or `useTypeaheadSearch` because they depend on WheelPicker's internal contract.

## Common Pitfalls

### Pitfall 1: Children Prop Typing Conflict
**What goes wrong:** Adding `children?: Snippet` to `WheelPickerWrapperProps` and then spreading `WheelPickerWrapperProps` as the component's props type causes TypeScript to complain that `children` is redeclared.
**Why it happens:** Svelte 5 injects `children` as a compile-time reserved prop for components with `{@render children()}`. Declaring it manually creates a conflict with the compiler's implicit injection.
**How to avoid:** Keep `children` out of `WheelPickerWrapperProps`. Use an intersection at the component script level: `let { classNames, children }: WheelPickerWrapperProps & { children?: Snippet } = $props()`.
**Warning signs:** TypeScript error `Type 'Snippet' is not assignable to...` or `Duplicate identifier 'children'` in the component file.

### Pitfall 2: Stale dist After Source Changes
**What goes wrong:** Modifying `src/lib/index.ts` or adding `WheelPickerWrapper.svelte` then running `npm pack` without first running `pnpm run package` â€” the tarball contains the old `dist/` contents.
**Why it happens:** `prepack` runs `npm run package` automatically for `npm pack` but NOT for `pnpm pack`. The `package.json` `"prepack"` hook is triggered by npm lifecycle, not pnpm.
**How to avoid:** Always run `pnpm run package` explicitly before testing the output, OR use `npm pack` (not `pnpm pack`) which respects the `prepack` hook. The `package` script runs both `svelte-package` and `publint` in sequence.
**Warning signs:** `dist/index.js` still shows old exports after editing `src/lib/index.ts`.

### Pitfall 3: publint Failure After Export Changes
**What goes wrong:** After removing hook exports from `index.ts`, publint may warn if `dist/index.d.ts` still references removed exports (stale dist).
**Why it happens:** publint reads the actual `dist/` files that svelte-package generated, not the source. If dist is stale, it will report mismatches.
**How to avoid:** Run `pnpm run package` after every source change. This always regenerates `dist/` from scratch.
**Warning signs:** `publint` output shows `Exports defined in "exports" but not found in entry` or similar.

### Pitfall 4: WheelPickerWrapper in vitest browser project include list
**What goes wrong:** Adding `WheelPickerWrapper.test.ts` to the vitest config but omitting it from the browser project's `include` array means the test runs in node environment without DOM, causing `container` or `render()` failures.
**Why it happens:** `vitest.config.ts` uses two projects (unit/node, browser) with explicit `include` arrays. New test files are not auto-discovered â€” they must be added to the correct project.
**How to avoid:** Add `WheelPickerWrapper.test.ts` to the browser project's `include` list in `vitest.config.ts`.

### Pitfall 5: Generic Type Lost in dist Declaration
**What goes wrong:** `dist/WheelPicker.svelte.d.ts` shows `Component<WheelPickerProps<string>, {}, "">` â€” the generic parameter is hardcoded to `string`. Consumers cannot use `WheelPicker` with `number` values and get proper type inference.
**Why it happens:** `@sveltejs/package` generates the declaration from Svelte 5's component type system. The generic is preserved only when the component's script uses the generic form of `$props()`. Currently `WheelPicker.svelte` destructures with `: WheelPickerProps` (not `WheelPickerProps<T>`), which erases the generic.
**Impact for this phase:** Phase 5 should verify the current type output after rebuild and determine if full generic inference (accepting `T extends string | number`) is achievable. If the svelte-package output hardcodes `string`, this is a known limitation of Svelte 5's current declaration generation â€” document and accept, or investigate the `ComponentProps` utility type workaround.
**Warning signs:** `dist/WheelPicker.svelte.d.ts` shows `WheelPickerProps<string>` not `WheelPickerProps<T>`.
